# 개요
- 서버에 접근하는 클라이언트의 권한을 확인하는 방법
- Basic 과 Bearer 방식이 있음

# HTTP 통신 속 인증헤더
``` HTTP
Authorization: <type> <credentials>
```
- 올바른 인증정보라면 200 OK 코드와 함께 요청한 데이터가 돌아오고
- 인증헤더를 누락하면 401 Unauthorized
- 권한이 없는 접근이라면 403 Forbidden 에러가 돌아옴

# 인증 방식
## Basic 인증
``` http
Authorization: Basic base64({USERNAME}:{PASSWORD})
```
- 인증정보 : [[base64]]로 인코딩한 “ID:비밀번호” 문자열을 헤더에 입력
- 간단하고 쉽게 복호화 가능하기 때문에 꼭 [[HTTPS]], [[SSL/TLS]]로 통신해야 안전
- 단점 : 
	- 서버에 사용자 목록을 저장해서, 요청 리소스가 많거나 사용자가 많으면 서버 부담 커짐
	- 리소스 별 권한 설정을 위해서는 추가 구현이 필요함

## Bearer 인증
``` http
Authorization: Bearer <token>
```
- [[Oauth의 이해|OAuth 2.0]]에서 사용하는 토큰 인증 방식
- Bearer = 소유자  라는 뜻

> [!info] Bearer 토큰
> - 액세스 토큰으로 사용되는 토큰의 유형
> - 불투명한 문자열 - 사용자가 임의로 확인할 수 없는 데이터(클라이언트가 해석할 수 없어야 함)
> - 토큰의 형태는 16진수의 문자열일 수도, JWT일 수도 있음

### 장점
- 안전하고, 확장이 쉬우며 사용자가 늘어도 토큰 검증과정의 소요시간이 항상 같음
- 토큰 자체에 메타데이터를 가지고 있어서 서버는 토큰을 발급만 하고 보관할 필요가 없음

### 단점
- 토큰이 외부에 노출되면 다른 서비스도 바로 리소스를 접근할 수 있음
